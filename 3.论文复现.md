复现他人的工作是科研中的重要一步，其用处主要包括以下几点：

- **寻找新的idea：**科研不是平地起大厦，90%的科研工作都是在现有的工作上进行优化形成的。作为一个初学者，直接通过阅读论文来找到其方案设计中的不足是十分困难的，这时候，复现他人的工作就显得十分重要。复现的过程中，你可以分析其方案在效率、鲁棒性、问题建模、部署可行性上的不足，从而迸发出自己的idea。

- **复现也是学习的一部分：**对于科研入门的小白来说，复现别人的工作是一个非常好的学习的机会，在这个过程中，你会接触到实验环境配置、数据处理、库依赖冲突等一系列工程问题，并逐步理解代码与论文内容的对应关系。早踩坑早避坑，这些宝贵的经验能够为你未来的工作做铺垫。

- **将这个方法作为论文对比的baseline：**这一般涉及到论文的实验部分，既然都是需要完成的工作，为何不早点干呢？

明白了论文复现的重要性，下面我们通过一个实例来说明如何复现一篇论文的方案。

Step 1 找代码

需要注意一点，并不是所有的论文都会公开代码。一般来说，对自己工作有自信，或者优秀科研/企业单位（如微软、top级别的高校实验室）才会选择公开代码。大部分的论文工作都不会公开代码（毕竟谁想自己的工作再被别人鞭尸一次呢），这些工作的复现难度较大，需要读懂论文中的方案设计后自己编写代码（也可以直接email论文作者要求其提供代码，不过这种的成功率一般很低）。这里我们只介绍前者。

开源工作一般会在论文中提供其代码的github下载地址，一般来说，这个地址常常出现在论文的abstract章节、脚注、evaluation章节或者附录当中。以《Linking

Step2 准备工作

准备工作主要包括数据集下载以及运行环境配置。

先从数据集下载说起，一般来说，论文都会使用一些公开的数据集（coco、CIFA10、MNIST等）或者作者从仿真器、真实环境中采集到的数据，由于数据体量较大（通常达到上百MB甚至GB级别），这些文件并不会附在github代码仓库中，你需要自行从相关的网站上进行下载。目前常见的数据托管平台包括：

- HuggingFace（[Hugging Face -- The AI community building the
  future.](https://huggingface.co/)）：全球领先的开源机器学习平台，提供大量模型和数据集，许多优秀的模型工作都会在这个平台进行发布。

- Zenodo（[Zenodo](https://zenodo.org/)）：一个开放获取的数据存储平台，提供论文、数据集、技术报告等科研成果的托管，并且支持为每个上传项目分配唯一的持久标识符（DOI）以便于公开分享。

注意：上述展示的只是一个最简单的例子，并且由于代码维护的较好，整个复现过程，包括数据集的准备和代码的运行都没有遇到意外情况。请记住，你复现一个工作的难度完全取决于作者对于项目维护的好坏程度。无论是缺少必要的文件，还是readme文件写得一团糟，都会极大地影响你在复现过程中的体验。除此之外，复现过程中你还可能遇到各种意外情况，如依赖库版本冲突、硬件环境配置（例如，使用CPU还是GPU运行神经网络）不同，这些都可能导致程序报错，需要你逐一排查。错误排查这一块，你可以参考以下内容：

- 查看项目github的issue：issue为项目开发人员和用户提供了交流渠道，在这里，你可以查看其它用户对于项目bug的提问，并对照自己的项目环境进行问题排查。issue的使用可以参考[GitHub
  Issues 快速入门 - GitHub
  文档](https://docs.github.com/zh/issues/tracking-your-work-with-issues/learning-about-issues/quickstart)。

- 各类论坛：函数报错、库冲突问题等，可以在CSDN、Stake
  Overflow这类编程论坛上进行查询。

- 大模型：目前大模型的代码能力都非常强，用它来做debug是一个不错的选择。不过要注意的是，大模型解决问题的能力很大程度上仰赖于上下文，用它进行debug时，不要简单粗暴地直接将报错信息丢给它，而是尽量描述清楚你的问题，包括出错函数所在的库、库的版本、函数依赖等。

大多数情况下，复现别人的工作都是一个痛苦且折磨的事，并且有时候你会发现复现结果根本达不到论文中所声称的水平（这在AI领域非常常见）。因此，请优先考虑复现那些更有可能复现成功的工作。听起来像是玄学，但根据我的经验来看，具备以下特征的项目往往复现成功的概率更大：

- 来自国内外顶级科研机构，如微软、DeepMind

- 来自特定研究领域内的顶级实验室或大牛，这需要你对本领域的科研圈子有大致的了解

- Github stars/forks数目较多

需要注意的是，复现不仅仅是再现别人的工作，读懂和理解代码也是复现工作的重要一环。这个过程也需要你有所侧重，例如，几乎所有的项目代码都包括参数的初始化、数据的导入/导出等功能模块，这些内容都只需要一笔带过，你应该重点关注论文核心方法对应的代码实现，理清其创新点的实现过程。

最后，复现也要形成自己的笔记or工作记录，主要记录复现出现的问题及其解决方案，防止后续需要迁移所导致"从头再来"的问题。笔记工具推荐Notion、语雀。
